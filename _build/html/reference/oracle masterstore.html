Oracle : MasterStore¶

The Oracle Setup will be performed by Customer : 1. The DB will reside on new Hardware. 2. The DB Server will be set up in their own network segment. 3. Planned are 4 DB Servers. Two of the Servers will be set up in one of two data centres. The two Servers per DC will be set up as RAC and based on VPLEX Storage 4. One RAC implementation is used for the PROD, the other for the TEST environment. 5. The RAC Version will be 11.2.0.4 6. All standard settings will be used concerning Linux User and Linux rights, ASM Drives, Storage Groups, Backup & Recovery concepts and processes for Oracle, network layout, naming conventions, filesystems, UNIX version, etc. 7. The DB will use a Unicode charset. The name of the charset is AL32UTF8. 8. Table partitioning and PL-/SQL will be used. No special Oracle features are necessary. Standard Oracle features are: 1. Partitioning 2. Real Application Clusters 3. Automatic Storage Management 4. OLAP 5. Data Mining 6. Real Application Testing 9. The Tablespace layout will be chosen by the DBAs. Default tablespaces for the DB users is used. 10. CS Vision needs three DB Schema: 1. Deployment User: For software installation, software updates, and performance analysis by CS Vision. DB Objects are stored here. 2. Application User: This User will be used by the application to connect to the DB. The application doesn’t use DDL statements and needs read access to the objects owned by the deployment user.The Application User doesn’t need tablespace of its own. 3. Support User: Is used from CS Vision to access the DB. He doesn’t use DDL Statements and read access to the Deployments User’s DB objects. The Support User doesn’t need tablespace of its own. 11. For the deployment of updates in the PROD environment, the DBAs need, in the sense of a Change process, a summary of all DB objects that will be changed, all new DB objects, the amount of data in change.Based on this information all necessary resources (especially tablespace) will be provided 12. The duration of DB procedures will be stored within DB tables by the application on behalf of performance measures. Every other DB monitoring is provided by the DBAs. 13. The 5 application server nodes in one DC will use (configurable) 50 permanent JDBC connections. The connections will be established during the startup of the application. In case of a DB restart, the application server will perform an automatic reconnect (automatic fail-over). The behaviour will be tested on the cluster after the PROD system is set up. Create various DB objects in Oracle The following will be discussed together with the Customer’s DBA team and is subject to change based on their existing standards and processes. Prerequisite: 1. A new DB on its own Hardware will be installed. 2. The DB Servers will run on their own network segment. 3. Planned are 4 DB-Servers, 2 in each Datacenter as RAC, based on VPLEX Storage. 4. With one RAC for PROD, one for TEST each in in a different Datacenter. 5. Infrastructure will be according to existing standards (Customer), i.e., Linux User and Rights, ASM Drives and Diskdrive groups, Backup and Recovery concept and processes, network layout, naming conventions, filesystem, Unix version. 6. The system will use Table Partitioning and PL/SQL procedures. There would also be some java based stored procedures. 7. The tablespace layout can be selected be the DBAs without any restrictions. Default tablespaces of the DB-Users will be used. 8. The 5 application servers use about 50 JDBC connections against the DB. This is configurable. If the DB is restarted, the application servers perform automatic reconnects against the DB (automatic fail-over). This behaviour will be verified with cluster tests since CS doesn’t have a RAC system to test this. 9. The execution times of DB procedures are logged in DB-Tables. This is used for performance measures from the application. Every other form of DB monitoring is supplied from the DBAs. 10. No other Oracle specials are necessary, no extra installation. Standard installation includes: 1. Partitioning 2. Real Application Clusters 3. Automatic Storage Management 4. OLAP 5. Data Mining 6. Real Application Testing Schemas : The schemas required will be as follows : 1. Deployment User – Used for installation and updates 1. This user will be the owner of the application schema and will perform the following actions: 1. Software Installation 1. Creating tables, procedures, indexes, views, functions, etc. 2. Software Updates 1. Altering existing objects, creating new objects and some data updates. 3. Performance analysis 2. Permissions 1. Connect to the database (create session) 2. Alter session related parameters (alter session) 3. Create objects (table, views, indexes, procedures and functions) 2. Application User – Used by the Application to access the database 1. This user will have DML rights on the Deployment User’s schema and the Pentaho User’s schema and will not need any additional space. 2. Permissions 1. Connect to the database (create session) 2. Alter session related parameters (alter session) 3. Grant Object Level Permissions for tables and views 1. Select 2. Insert 3. Update 4. Delete 4. Grant Execute Permissions 1. Procedures 2. Functions 3. Readonly User – Used by the Support to access the database 1. This user will have read access on the Deployment User’s schema and the Pentaho User’s schema in order to run some queries to check for issues in the data. 2. Permissions on Application Schema & Pentaho Schema 1. Connect to the database (create session) 2. Grant SELECT permissions on all Tables & Views The Deployment user will connect to the schema and run the SQL scripts to create various objects. Thereafter the user will grant permissions on these objects to the application user. 1. CS will perform the following checks: 1. The tables and procedures have been created properly. 2. CS will run some select scripts to check if the updates have been successful.
